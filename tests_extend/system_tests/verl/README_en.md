# Qwen3-30B-A3B

## Overview

This document provides an example of training Qwen3-30B-A3B using verl.

## Hardware Environment

| Hardware | Configuration |
| :--: | :-------------: |
| NPU | 32 x Ascend NPUs |

NOTE: This example uses two Atlas 800T A3 nodes for training. Under the current experimental configuration, two A2 nodes may experience out-of-memory (OOM).

## Software Environment

| Component | Version |

| CANN | 8.5.0 |
| torch | 2.8.0 |

| vLLM | 0.13.0 |

## Model Training

### 1. Prepare Data

Download the weights [Qwen/Qwen3-30B-A3B](https://huggingface.co/Qwen/Qwen3-30B-A3B/tree/main).

Convert the weights from Hugging Face format to Megatron Core format:

```bash
# The weight conversion script is located in the verl code repository
cd verl

python scripts/converter_hf_to_mcore.py \
    --hf_model_path /weight/Qwen3-30B-A3B \
    --output_path /weight/Qwen3-30B-A3B-dist \
    --use_cpu_initialization
```

Download the dataset [openai/gsm8k](https://huggingface.co/datasets/openai/gsm8k/tree/main).

### 2. Configure the training script

* First, adjust the following paths as needed:

`HF_MODEL_PATH`: Qwen3-30B-A3B path

`DIST_CKPT_PATH`: converted weight path

`data.train_files`, `data.val_files`: dataset paths

`trainer.default_local_dir`: path for saving checkpoints. This can be left unset when `trainer.save_freq=-1` (i.e., no checkpoints are saved). Ensure there is sufficient space under this path to store checkpoints.

* Next, adjust other parameters as needed. Here are a few examples:

`data.max_prompt_length` `data.max_response_length`: These are used to set the maximum number of tokens for the input prompt and the maximum token length for the response generated by the RL algorithm, respectively. Increasing them allows for processing longer prompts but increases memory consumption and computation time; decreasing them saves memory but causes sequences to be truncated, limiting the model's ability to generate complete answers. The values used in this example, 1024/2048, are empirical settings that achieve a good balance between sequence length coverage and memory/computation overhead under the current hardware resource constraints.

`actor_rollout_ref.actor.ppo_mini_batch_size` `actor_rollout_ref.actor.ppo_micro_batch_size_per_gpu`: These determine the total number of samples used in each PPO update and the batch size for a single forward pass, respectively. Constraint: the former must be divisible by the latter, and the latter must be less than or equal to the former. Under specific hardware resource constraints, maximize the latter as much as possible to improve training efficiency.

* Finally, you can add some advanced features supported by MindSpeed as needed, for example:

Using mbridge for online conversion of HuggingFace weights (mbridge package installation required)

```bash
actor_rollout_ref.actor.megatron.use_mbridge=True \
actor_rollout_ref.actor.megatron.use_dist_checkpointing=False \
actor_rollout_ref.ref.megatron.use_dist_checkpointing=False \
```

Use RoPE fusion optimization [RoPE fusion optimization](../../../docs/en/features/rotary-embedding.md)

```bash
+actor_rollout_ref.actor.megatron.override_transformer_config.position_embedding_type=rope \
+actor_rollout_ref.actor.megatron.override_transformer_config.use_fused_rotary_pos_emb=True \
```

### 3. Start training

The training script is `qw3-30bmoe-grpo-2node_base.sh`. Configure all required parameters in the training script, and then execute the training script on **all nodes**.

```bash
# # Ensure the training script is executed in the verl directory
cd verl

# # Execute the training script in the background and save the logs
nohup bash qw3-30bmoe-grpo-2node_base.sh > qw3-30bmoe-grpo-2node_base.log 2>&1 &
tail -f qw3-30bmoe-grpo-2node_base.log
```
